# Границы открытой публикации

## Назначение

Раздел определяет, какие материалы проекта можно публиковать в открытом GitHub-репозитории, какие требуют редактирования и какие должны оставаться внутренними. Эта рамка нужна для того, чтобы книга могла быть публичной научно-методической работой, не раскрывая закрытую инфраструктуру, персональные данные и организационные материалы без разрешения.

## Принцип публикации

Открытая публикация должна отвечать двум условиям:

1. Материал помогает понять методологию, архитектуру или ход разработки платформы.
2. Материал не раскрывает сведения, которые могут нарушать права участников, внутренние правила организации, безопасность платформы или требования к персональным данным.

## Материалы, которые можно публиковать открыто

| Тип материала | Условие публикации | Пример |
|---|---|---|
| Академический текст | Без утверждений, требующих юридического подтверждения. | Обзор проекта, исследовательский контекст, статьи. |
| Методическая модель | Если компетенции и задания являются авторскими или разрешены к публикации. | ПК-ИИ-1 — ПК-ИИ-10, учебная карта, ФОС. |
| Архитектурные схемы | На уровне доменной модели, без секретов рабочей инфраструктуры. | Learning, Assessment, Certification, Storage, Audit. |
| Синтетические данные | Если они не совпадают с реальными персональными данными. | Демонстрационный JSON сертификата. |
| Ноутбуки | Если используют относительные пути и открытые/синтетические данные. | Модель жизненного цикла сертификата. |
| Библиография | При корректном указании источников. | Нормативные, методические и технические источники. |

## Материалы, требующие редактирования

| Тип материала | Что проверить перед публикацией |
|---|---|
| Скриншоты интерфейса | Удалить ФИО, e-mail, токены, внутренние URL, идентификаторы пользователей. |
| Логи и отчеты ошибок | Удалить IP-адреса, пути на сервере, трассировки ошибок рабочей инфраструктуры, идентификаторы сессий. |
| Метрики обучения | Анонимизировать и агрегировать, исключить возможность восстановления личности. |
| Примеры документов | Использовать синтетические ФИО, даты, номера и QR-коды. |
| Фрагменты локальных актов | Публиковать только после разрешения организации. |
| Код интеграций | Удалить ключи, токены, адреса сервисов и сведения о закрытой инфраструктуре. |

## Материалы, которые не следует публиковать

- ключи доступа, токены, `.env`, пароли, OAuth-настройки;
- реальные персональные данные слушателей, преподавателей и администраторов;
- реальные работы слушателей без разрешения и обезличивания;
- дампы рабочей базы данных и файлового хранилища;
- закрытые внутренние регламенты, если нет разрешения на публикацию;
- сведения о неустраненных уязвимостях и деталях защиты;
- документы, в отношении которых неясны авторские права или лицензионные ограничения;
- материалы, создающие впечатление утвержденного юридического статуса документа без решения образовательной организации.

## Чек-лист перед добавлением файла в книгу

Перед публикацией нового файла нужно ответить на вопросы:

1. Есть ли в файле реальные персональные данные.
2. Есть ли ключи, токены, закрытые URL или параметры инфраструктуры.
3. Ясно ли, кому принадлежат авторские права на материал.
4. Можно ли заменить реальные данные синтетическими.
5. Не содержит ли текст юридических утверждений без источника и согласования.
6. Обновлен ли `_toc.yml`, если файл является главой книги.
7. Проходит ли файл проверку `scripts/validate_data.py`, если это JSON/YAML.
8. Собирается ли книга после добавления файла.

## Публикационная позиция проекта

Открытая книга должна демонстрировать ход исследования и разработки: логику курса, архитектуру платформы, модель данных, публичные диаграммы, воспроизводимые ноутбуки и библиографическую базу. При этом она не должна становиться зеркалом рабочей информационной системы. Граница между «публичным научным описанием» и «внутренней эксплуатационной информацией» должна сохраняться на протяжении всего проекта.
