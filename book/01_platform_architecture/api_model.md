# API-модель

## Назначение раздела

API-модель описывает публично допустимую логику взаимодействия между пользовательским интерфейсом, backend-слоем, хранилищем, модулем оценивания и модулем документа об обучении. Этот раздел не раскрывает реальные адреса сервисов, секреты, ключи, внутренние правила маршрутизации и параметры production-инфраструктуры.

## Группы API

| Группа | Примеры операций | Публичный уровень описания |
|---|---|---|
| Courses | Получить курс, модули, уроки, ресурсы. | Можно описывать структуру ресурсов и статусы доступности. |
| Enrollment | Записать слушателя, получить прогресс, завершить курс. | Можно показывать жизненный цикл без персональных данных. |
| Assessment | Создать попытку, отправить работу, проверить, выставить результат. | Можно описывать статусы и критерии проверки. |
| Competencies | Получить покрытие компетенций, зафиксировать подтверждение. | Можно показывать связь ПК-ИИ-1 — ПК-ИИ-10 с заданиями. |
| Certification | Сгенерировать документ, изменить статус, отозвать, перевыпустить. | Можно описывать доменную логику, но не утверждать юридический статус. |
| Verification | Проверить документ по QR-коду или коду проверки. | Можно показывать минимальный публичный ответ. |
| Storage | Загрузить ресурс, получить ссылку, проверить хеш. | Нельзя раскрывать закрытые URL и политики доступа. |
| Audit | Зафиксировать значимое действие. | Можно описывать типы событий без реальных логов. |

## Пример публичного контракта

Пример ответа публичной QR-проверки должен быть минимальным:

```json
{
  "status": "valid",
  "documentType": "project_training_document",
  "programTitle": "ИИ-инструменты в преподавательской, научной и повседневной работе",
  "hours": 72,
  "issuedAt": "2026-05-30",
  "registrationNumber": "FEM-DPO-AI-2026-0001"
}
```

Даже этот пример является синтетическим. Реальный публичный ответ должен быть согласован с образовательной организацией, особенно если в нем предполагается отображать ФИО слушателя, период обучения, статус документа или сведения о подписанте.

## Статусы и ошибки

Для публичной модели достаточно фиксировать ограниченный набор статусов:

- `draft` — документ подготовлен, но не выдан;
- `issued` — документ выдан;
- `revoked` — документ отозван;
- `reissued` — документ перевыпущен;
- `not_found` — проверочный код не найден;
- `invalid` — документ не является действительным.

Внутренние причины ошибок, служебные трассировки и сообщения инфраструктуры не должны попадать в публичный API-ответ.

## Принцип минимального раскрытия

API должен возвращать только те данные, которые нужны для конкретного сценария. Для публичной проверки документа это особенно важно: QR-код не должен превращаться в открытый реестр персональных данных. В публичной книге можно описывать принцип работы проверки, но нельзя публиковать реальные проверочные коды, персональные записи слушателей и production-адреса.

## Связь с ветками проекта

- Ветка 11 задает общий контур платформенной интеграции курса ДПО.
- Ветка 10 содержит `certificate_api_contracts.json` и модель публичной проверки документа.
- Ветка 5 задает операции, связанные с оценочными элементами, попытками, проверками и статусами.

API-модель должна обновляться после каждого существенного изменения методической модели курса: добавления новой компетенции, изменения фонда оценочных средств, изменения статусов завершения или пересмотра юридического статуса документа.
