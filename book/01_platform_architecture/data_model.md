# Модель данных

## Назначение раздела

Раздел описывает публичную доменную модель платформы. Его задача — показать, какие сущности нужны для реализации курса ДПО, оценивания, фиксации прогресса, документа об обучении и аудита, не раскрывая при этом внутреннюю схему базы данных, закрытые поля, персональные данные и детали инфраструктуры.

Платформенная интеграция курса ДПО представлена в ноутбуке [](architecture_overview.ipynb). Сертификационная модель данных вынесена в раздел [](../05_certification/certificate_data_model.ipynb).

## Доменные области

| Домен | Назначение | Ключевые сущности |
|---|---|---|
| Learning | Описание курса, модулей, уроков и материалов. | `CourseTemplate`, `CourseRun`, `Module`, `Lesson`, `CourseResource` |
| Enrollment | Связь слушателя с запуском курса и его прогрессом. | `Enrollment`, `ProgressState`, `CompletionStatus` |
| Assessment | Задания, попытки, проверки и результаты. | `AssessmentItem`, `AssignmentSubmission`, `Grade`, `Rubric` |
| Competency | Трассировка компетенций ПК-ИИ-1 — ПК-ИИ-10. | `Competency`, `CompetencyEvidence`, `CompetencyResult` |
| Certification | Шаблон и выданный документ. | `CertificateTemplate`, `IssuedCertificate`, `VerificationCode` |
| Storage | Файлы, PDF, изображения и учебные ресурсы. | `StorageObject`, `FileHash`, `AssetReference` |
| Access Control | Роли и права. | `User`, `Role`, `Permission`, `CourseAssignment` |
| Audit | История значимых действий. | `AuditLog`, `AuditEventType`, `ActorRef` |

## Логика связей

Курс начинается с `CourseTemplate`, который описывает устойчивую структуру программы. Когда программа запускается для конкретной группы слушателей, создается `CourseRun`. Слушатель связывается с запуском через `Enrollment`. Учебные материалы и задания привязаны к урокам и модулям, а результаты выполнения заданий фиксируются через `AssignmentSubmission` и `Grade`.

Компетенции ПК-ИИ-1 — ПК-ИИ-10 не должны храниться как декоративный список в конце программы. Они должны быть связаны с проверяемыми артефактами: заданием, критерием, результатом проверки и итоговым статусом. Такая модель позволяет объяснить, почему слушателю может быть выдан документ, какие компетенции подтверждены и какие элементы курса стали основанием для этого решения.

Документ об обучении создается только после прохождения условий завершения курса. В технической модели `IssuedCertificate` хранит ссылку на слушателя, запуск курса, шаблон документа, статус, регистрационный номер, код проверки и объект PDF-хранения. PDF не должен быть единственным источником истины: он является представлением данных, а не заменой структурированной записи.

## Публичная и внутренняя модели

Публичная модель описывает сущности и связи на уровне, достаточном для научного отчета. Внутренняя модель может включать дополнительные поля: технические идентификаторы, служебные статусы, индексы, параметры интеграции, журналы ошибок и настройки инфраструктуры.

| Можно публиковать | Нельзя публиковать без отдельного решения |
|---|---|
| Обобщенные сущности и связи. | Реальные персональные данные слушателей и сотрудников. |
| Синтетические примеры JSON/YAML. | Закрытые ключи, токены, `.env`, внутренние URL. |
| Диаграммы доменной модели. | Полные дампы базы данных и production-логи. |
| Анонимизированные показатели качества. | Материалы, содержащие сведения ограниченного доступа. |

При проектировании публичных примеров необходимо учитывать требования к обработке и распространению персональных данных {cite}`personal_data_law_152fz`.

## Проверяемость и воспроизводимость

Для публикационной версии книги важно сохранять структурированные файлы: YAML, JSON, CSV и notebooks. Они позволяют читателю видеть не только итоговый текст, но и исходную структуру решений. Поэтому данные, которые можно раскрывать, должны публиковаться как данные, а не превращаться в пересказ.

## Текущий статус

Модель данных является проектной. Она задает основу для разработки платформы и для дальнейшего обсуждения с образовательной организацией, но не является утвержденной внутренней схемой информационной системы.
