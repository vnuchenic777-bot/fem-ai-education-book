# Модель данных

## Назначение раздела

Раздел описывает публичную доменную модель платформы. Его задача — показать, какие сущности нужны для реализации курса ДПО, оценивания, фиксации прогресса, документа об обучении и аудита, не раскрывая при этом внутреннюю схему базы данных, закрытые поля, персональные данные и детали инфраструктуры.

Платформенная интеграция курса ДПО представлена в ноутбуке [](architecture_overview.ipynb). Сертификационная модель данных вынесена в раздел [](../05_certification/certificate_data_model.ipynb).

## Доменные области

| Домен | Назначение | Основные объекты публичного описания |
|---|---|---|
| Обучение | Описание курса, модулей, уроков и материалов. | Программа курса, запуск курса, модуль, урок, учебный ресурс. |
| Зачисление и прогресс | Связь слушателя с запуском курса и динамикой прохождения. | Зачисление, состояние прогресса, статус завершения. |
| Оценивание | Задания, попытки, проверки и результаты. | Элемент контроля, отправленная работа, критерии, результат проверки. |
| Компетенции | Трассировка компетенций ПК-ИИ-1 — ПК-ИИ-10. | Компетенция, подтверждающий артефакт, итоговый результат. |
| Документ о результате обучения | Шаблон, выпуск, статус и проверка документа. | Шаблон документа, выданный документ, код проверки. |
| Хранение материалов | Файлы, изображения, учебные ресурсы и подтверждающие документы. | Файл, ссылка, контрольная сумма, ссылка на ресурс. |
| Доступ | Роли и права участников. | Пользовательская роль, разрешение, назначение на курс. |
| Аудит | История значимых действий. | Событие, тип действия, участник действия. |

## Логика связей

Курс начинается с устойчивого описания программы: целей, модулей, уроков, материалов и форм контроля. Когда программа запускается для конкретной группы слушателей, появляется отдельный запуск курса. Слушатель связывается с этим запуском через запись о зачислении. Учебные материалы и задания привязаны к урокам и модулям, а результаты выполнения заданий фиксируются через отправленные работы, проверки и оценки.

Компетенции ПК-ИИ-1 — ПК-ИИ-10 не должны храниться как декоративный список в конце программы. Они должны быть связаны с проверяемыми артефактами: заданием, критерием, результатом проверки и итоговым статусом. Такая модель позволяет объяснить, почему слушателю может быть выдан документ, какие компетенции подтверждены и какие элементы курса стали основанием для этого решения.

Документ об обучении создается только после прохождения условий завершения курса. В проектной модели для него фиксируются связь со слушателем и запуском курса, шаблон, статус, регистрационный номер, код проверки и файл печатного представления. PDF не должен быть единственным источником истины: он является представлением данных, а не заменой структурированной записи.

## Публичная и внутренняя модели

Публичная модель описывает сущности и связи на уровне, достаточном для научного отчета. Внутренняя модель может включать дополнительные поля: технические идентификаторы, служебные статусы, индексы, параметры интеграции, журналы ошибок и настройки инфраструктуры.

| Можно публиковать | Нельзя публиковать без отдельного решения |
|---|---|
| Обобщенные сущности и связи. | Реальные персональные данные слушателей и сотрудников. |
| Синтетические примеры JSON/YAML. | Закрытые ключи, токены, `.env`, внутренние URL. |
| Диаграммы доменной модели. | Полные дампы базы данных и журналы рабочей инфраструктуры. |
| Анонимизированные показатели качества. | Материалы, содержащие сведения ограниченного доступа. |

При проектировании публичных примеров необходимо учитывать требования к обработке и распространению персональных данных {cite}`personal_data_law_152fz`.

## Проверяемость и воспроизводимость

Для публикационной версии книги важно сохранять структурированные файлы: YAML, JSON, CSV и notebooks. Они позволяют читателю видеть не только итоговый текст, но и исходную структуру решений. Поэтому данные, которые можно раскрывать, должны публиковаться как данные, а не превращаться в пересказ.

## Публикационное ограничение

Модель данных является проектной и представлена на уровне, достаточном для научно-методического описания. Она задает основу для разработки платформы и дальнейшего обсуждения с образовательной организацией, но не является утвержденной внутренней схемой информационной системы.
