## Назначение раздела

Раздел описывает фонд оценочных средств для программы повышения квалификации «ИИ-инструменты в преподавательской, научной и повседневной работе: от понимания принципов до практического применения». В рамках общего проекта он раскрывает методическую модель проверки результатов обучения: текущий контроль, промежуточный контроль по модулям, задания-артефакты, тестовые проверки, оценивание по критериям, порядок доработки, повторную сдачу, итоговый проект и асинхронную защиту.

Фонд оценочных средств не рассматривается как приложение к курсу, которое составляется после написания лекций. В данном проекте он является управляющим слоем: именно он определяет, какие действия должен выполнить слушатель, какие продукты должны быть загружены в платформу, какие статусы фиксируются в системе и при каких условиях компетенция считается освоенной.

## Место раздела в общей архитектуре проекта

Фонд оценочных средств соединяет три уровня проекта.

Первый уровень — образовательный. Курс формирует 10 профессиональных компетенций ПК-ИИ-1 — ПК-ИИ-10. Каждая компетенция подтверждается двумя элементами: артефактом и тестовой или практической проверкой.

Второй уровень — организационный. Проверка выполняется преподавателем курса или назначенным проверяющим. ИИ может использоваться как помощник проверяющего, однако итоговый и спорный результат по программе ДПО должен подтверждаться человеком.

Третий уровень — цифровое сопровождение. Платформа должна хранить контрольные элементы, попытки, проверки, оценки, комментарии, итоговые статусы и данные для документа об обучении. В публичной версии эти объекты описываются на уровне образовательной логики; подробные технические идентификаторы сохраняются в структурированных источниках проекта.

Подробные схемы связей представлены в вычислительных тетрадях:

- [notebook_01.ipynb](notebook_01.ipynb) — матрица компетенций, модулей и форм контроля;
- [notebook_02.ipynb](notebook_02.ipynb) — жизненный цикл проверки и техническая модель оценивания.

## Постановка задачи

Главная задача ветки — разработать такую модель оценивания, которая позволяет подтвердить не только знание слушателя о возможностях ИИ, но и способность выполнить профессиональное действие: поставить задачу, сформулировать запрос, проверить результат, исправить ошибки, подготовить учебный или методический материал, разработать задание с обязательным использованием ИИ студентами и обосновать его внедрение в дисциплину.

Проблема заключается в том, что обычный тест не может подтвердить профессиональную готовность преподавателя к применению ИИ в образовательном процессе. Тест показывает знание терминов и процедур, но не показывает, способен ли слушатель создать применимый материал. Поэтому в модели используется двухчастное доказательство освоения компетенции: артефакт и проверка знаний или навыков.

## Методическая модель

### Двухчастное подтверждение компетенции

Каждая компетенция считается освоенной только при выполнении двух условий:

1. Слушатель создал и сдал артефакт.
2. Слушатель прошёл тестовую или практическую проверку.

Артефакт показывает способность создать профессиональный продукт. Тест или практическая проверка показывает понимание понятий, процедур, ограничений и типовых ситуаций.

### Формы контроля

Основные формы контроля представлены в структурированных данных `data/structured_data.yaml` и `data/structured_data.json`.

| Форма контроля | Назначение |
|---|---|
| Текущее практическое задание | Проверка профессионального действия по модулю |
| Тестовая проверка модуля | Проверка понятий, процедур и распознавания ситуаций |
| Практическая проверка навыка | Проверка применения правила к новой задаче |
| Итоговый проект | Проверка интеграции всех компетенций в дисциплину слушателя |
| Асинхронная видео-защита | Проверка самостоятельного понимания логики проекта |

### Принцип оценивания артефактов

Артефакт оценивается не по внешней завершённости текста, а по качеству профессионального действия. Для большинства артефактов применяются следующие критерии:

- постановка задачи;
- структура запроса;
- наличие итераций;
- проверка результата;
- практическая применимость;
- соблюдение ограничений и требований безопасности.

Эти критерии позволяют отличить осмысленное использование ИИ от механического копирования машинного ответа.

## Проектная модель фиксации результата

### Объекты платформенной фиксации

| Объект | Назначение |
|---|---|---|
| Контрольный элемент | Задание, тест, итоговый проект или защита. |
| Схема оценивания | Тип результата, максимальный балл, проходной порог и критерии. |
| Отправленная работа | Общая запись работы слушателя по контрольному элементу. |
| Версия отправки | Конкретная версия загруженной работы или ответа. |
| Проверка работы | Решение проверяющего, комментарий, статус и дата. |
| Оценка | Итоговый результат по заданию, модулю или курсу. |
| Документ о результате обучения | Запись о документе и подтвержденных компетенциях. |

### Жизненный цикл работы

Работа слушателя проходит следующие состояния:

1. Черновик.
2. Отправлено.
3. На проверке.
4. Требуется доработка.
5. Повторная отправка.
6. Принято или не принято.

Эта логика позволяет хранить историю попыток и отделять факт отправки от решения проверяющего. В случае доработки новая версия работы не должна затирать предыдущую. Это важно для прозрачности проверки и для анализа качества прохождения курса.

### ИИ-помощная проверка

ИИ-помощная проверка может выполнять предварительный анализ открытого ответа или артефакта:

- сопоставлять работу с критериями;
- выделять сильные и слабые стороны;
- предлагать предварительный комментарий;
- указывать на возможные пропуски.

ИИ-помощная проверка не должна самостоятельно присваивать официальный итоговый результат по ДПО. Преподаватель или проверяющий подтверждает, изменяет или отклоняет предварительное предложение.

## Требования к цифровой фиксации ФОС

### Функциональные требования

ФОС должен поддерживаться в платформе через следующие возможности:

1. Создание контрольных элементов разных типов: задание, тест, итоговый проект, рефлексия, защита.
2. Назначение схемы оценивания для каждого контрольного элемента.
3. Поддержка форматов результата: зачтено/не зачтено, баллы, статус, оценивание по критериям, ИИ-помощная проверка.
4. Хранение всех попыток отправки работы.
5. Хранение комментариев проверяющего и итогового решения.
6. Поддержка статуса «требуется доработка».
7. Возможность повторной отправки работы.
8. Отдельная фиксация освоения компетенции.
9. Формирование итогового статуса прохождения курса.
10. Подготовка данных для документа об обучении.

### Нефункциональные требования

1. Слушатель не должен видеть административные разделы и чужие работы.
2. Преподаватель должен видеть только те работы, которые относятся к назначенному запуску курса.
3. Изменение оценки должно попадать в журнал действий.
4. ИИ-помощная проверка не должна автоматически становиться официальным результатом.
5. Все даты должны храниться в едином системном формате и отображаться с учётом часового пояса организации или пользователя.

### Критерии готовности

Цифровая фиксация ФОС считается готовой, если:

- создана модель контрольных элементов;
- для каждого модуля можно создать артефакт и тест;
- слушатель может отправить работу;
- проверяющий может принять работу, вернуть на доработку или не принять;
- история попыток сохраняется;
- тесты с закрытыми вопросами проверяются автоматически;
- ИИ-помощная проверка требует подтверждения человеком;
- компетенция считается освоенной только при принятии артефакта и теста;
- итоговый статус курса формируется по всем 10 компетенциям;
- данные готовы для формирования документа об обучении.
